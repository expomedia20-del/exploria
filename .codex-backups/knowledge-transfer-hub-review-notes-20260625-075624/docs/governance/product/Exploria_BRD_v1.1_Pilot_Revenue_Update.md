# EXPLORIA - BRD v1.1 Pilot Revenue Update

# 1. کنترل سند، تغییر نسخه و قفل دامنه

این نسخه، BRD v1.0 را از یک سند ادغام‌شده کسب‌وکاری به نسخه‌ای آماده برای پایلوت‌های درآمدزای واقعی ارتقا می‌دهد. تمرکز v1.1، ورود رسمی Commercial Pilot Control Layer و قفل شدن ۲۶ محور کنترلی است تا هیچ موضوع حیاتی در BRD، FRD، MVP/MVCP، Product Backlog و Runbook از قلم نیفتد.

| نسخه | تغییر کلیدی | وضعیت |
| --- | --- | --- |
| v1.0 | ادغام اولیه BRD، تثبیت Vision، Scope، Stakeholders، MVP و مسیر FRD | Baseline قبلی |
| v1.1 | افزودن Pilot Revenue Portfolio، MVCP، Commercial Pilot Control Layer و Traceability به CPL-01..CPL-26 | این سند |
| v1.2 پیشنهادی | پس از دریافت Venue Profile کامل برج میلاد و عددگذاری مالی/عملیاتی | آتی |

| قانون قفل | شرح |
| --- | --- |
| قفل محتوایی | تمام محورهای CPL-01 تا CPL-26 باید در BRD v1.1 و FRD v1.1 دیده شوند. |
| حذف یا کاهش دامنه | فقط با Change Request رسمی، دلیل، اثر بر زمان/هزینه/ریسک و تصمیم مالک پروژه مجاز است. |
| Traceability | هر Requirement، User Story و ماژول FRD باید به BRD و حداقل یک CPL قابل ردیابی باشد. |

# 2. خلاصه مدیریتی نسخه v1.1

پروژه اکسپلوریا در این نسخه از «MVP نرم‌افزاری» به «پایلوت تجاری قابل سنجش» منتقل می‌شود. بنابراین هدف صرفاً ساخت چند ماژول نرم‌افزاری نیست؛ هدف، اجرای یک چرخه کامل توجه، اسکن، مأموریت، پاداش، مراجعه، داده، گزارش و درآمد در مکان واقعی است.

# 3. تعریف Scope: MVP، MVCP، Demo و Product

برای جلوگیری از Scope Creep، چهار سطح خروجی باید تفکیک شوند: نسخه نمایشی برای تصمیم‌سازی، MVP نرم‌افزاری برای حداقل قابلیت فنی، MVCP برای پایلوت تجاری در میدان، و Product نهایی برای توسعه مقیاس‌پذیر.

| MVP نرم‌افزاری | حداقل محصول فنی قابل استفاده | ثبت کاربر، QR، مأموریت، پاداش، داشبورد پایه | PWA/Web App، پنل Admin، پنل Merchant پایه |
| MVCP تجاری | حداقل پایلوت تجاری قابل اجرا در Venue واقعی | اثبات تعامل، مراجعه، درآمد و داده | سفیر، QR نصب‌شده، رسانه، کسب‌وکار، گزارش روزانه، Decision Gate |

# 4. پورتفوی پایلوت‌های اولویت‌دار درآمدزا

نسخه v1.1 پایلوت‌ها را به صورت پورتفوی درآمدزا می‌بیند، نه پروژه‌های جداگانه. هر Venue باید سناریوی مستقل داشته باشد اما همه از یک هسته مشترک داده، کمپین، پاداش و گزارش استفاده کنند.

# 5. مدل Venue / Zone / Hub / Touchpoint

اکسپلوریا باید هر مکان را به یک نقشه عملیاتی قابل پیکربندی تبدیل کند. در این مدل، Venue کل مجموعه است؛ Zone محدوده اجرایی؛ Hub نقطه ارزش؛ Touchpoint نقطه تعامل؛ Media Point نقطه جذب توجه؛ Merchant Node نقطه تبدیل تعامل به خرید؛ و Reward Point نقطه مصرف پاداش است.

| لایه | شرح کسب‌وکاری | اثر بر FRD |
| --- | --- | --- |
| Venue | کل مکان میزبان مانند اکوپارک، ارم یا برج میلاد | Entity اصلی برای پیکربندی، گزارش و قرارداد |
| Zone | محدوده پایلوت یا فاز اجرایی داخل Venue | فعال‌سازی محدود و کم‌ریسک پایلوت |
| Hub | مرکز ارزش مانند رواق، فودکورت، گنبد، زون بازی، سکوی دید یا فروشگاه | تعریف سناریو و KPI اختصاصی |
| Touchpoint | QR، استند، ورودی، مسیر، محل دریافت سرنخ یا ثبت مأموریت | اتصال رویدادهای کاربر به مکان |
| Media Point | کوله‌پشتی هولوگرافیک، نمایشگر ثابت یا رسانه محیطی | ردیابی اثر رسانه بر اسکن و مشارکت |
| Merchant Node | کسب‌وکار مشارکت‌کننده مثل رستوران، فروشگاه یا غرفه | درآمد، پاداش، مراجعه و گزارش فروشگاهی |
| Reward Point | محل یا روش مصرف امتیاز، تخفیف یا جایزه | کنترل مصرف و Attribution |

# 6. مدل تجاری، قرارداد و تقسیم درآمد

تا زمانی که مسیر درآمد، سهم طرفین، مالکیت داده و سازوکار تسویه روشن نشود، پروژه در مذاکره با میزبان، کسب‌وکار و اسپانسر به مشکل می‌خورد. BRD v1.1 این بخش را به عنوان الزام قفل‌شده وارد دامنه می‌کند.
